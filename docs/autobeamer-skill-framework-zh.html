从论文到高质量 Beamer Deck 的自动化技能框架

AutoBeamer 现在不是一个单一的“做幻灯片”流程，而是一个模式感知的技能系统： 先判断学习哲学，再选择创建、布局、构建、审查、TikZ 与验证技能，最后用可执行闸门防止规则漂移。

三种学习模式 被动学习、主动苏格拉底式学习、学术分享三条路径分流。
源文档优先 有 PDF 就先抽取文本和图，再考虑外部图片或重绘。
验证可执行 静态检查覆盖 overlay、References、appendix 和模式闸门。

主流水线

0 · Material Analysis 读论文、解析 PDF、抽取图像、梳理符号和先修知识。
1 · Needs Interview 只问无法安全推断的问题：时长、受众、模式、范围。
2 · Structure Plan 给出章节、页数、图像计划、符号计划和验证闸门。
3 · Draft 按 5 到 10 页批量写作，并在批次之间编译。
4 · Figures 源图、裁剪、TikZ、pgfplots、布局优化依次处理。
5 · Quality Loop XeLaTeX、静态验证、layout audit、visual check、模式审查。

框架一：三模式学习契约

模式不是样式参数，而是学习哲学。创建、审查、验证都必须先知道 deck 服务哪一种学习关系。

passive-study

被动学习模式

  • 面向跨领域初学者或领域先验较弱的研究者。
  • 目标是完整、耐心、自包含地讲清楚背景、符号、动机和例子。
  • 强调“降低不必要挫败”，但不降低数学和学术严谨度。
active-socratic

主动苏格拉底模式

  • 面向想自己重新发明知识的学习者。
  • 每个关键概念先给问题、尝试、提示，再给答案。
  • 适合推导、反例、思想实验、纸笔计算和反思问题链。
academic-presentation

学术分享模式

  • 面向 seminar、conference、defense、reading group 等现场分享。
  • 强调受众、时长、叙事主线、证据、References 和 backup slides。
  • 幻灯片是演讲提示，不是完整教材。

框架二：六个协作技能模块

每个技能只负责一个清晰边界。创建技能已经被压缩成 router，重内容放入 references，以减少上下文污染。

autobeamer-create

入口 router：选择模式、读取源文档、制定结构计划、串联后续技能。

autobeamer-layout

处理版式选择、DGV 语法、图文比例、稀疏页和溢出风险。

autobeamer-build

负责 XeLaTeX 编译、字体、依赖、日志和构建错误排查。

autobeamer-review

按三种模式加载不同 rubric，不再用同一把尺子评价所有 deck。

autobeamer-tikz

创建和审查 TikZ 图，保证数学准确性、可读性和 slide 尺寸适配。

autobeamer-validate

运行定量检查、静态闸门和 PDF 视觉检查，防止“看似通过”的假阳性。

亮点聚焦

这次重构真正有价值的地方，不是多写了几份文档，而是把过去容易漂移的隐性规则变成可搜索、可验证、可审查的工程契约。

Spotlight 01

学习哲学先于 Beamer 技术

  • 先确定学习关系，再决定写法、密度、练习和审查标准。
  • 旧的 Mentor / Presentation 只作为兼容别名，不再是主分类。
  • 主动苏格拉底模式被正式加入，而不是被动学习的变体。
Spotlight 02

源文档优先的图像策略

  • 给 PDF 时先运行 paper parser，抽取原文图片和结构信息。
  • 外部图片只能作为 fallback，必须本地保存并标注来源。
  • 数学图优先使用 TikZ 或重绘，保证符号和论证一致。
Spotlight 03

静态验证从建议变成工具

  • tools/validate_deck.py static 会拒绝 \pause、\onslide、\only、\uncover。
  • References 必须在 Thank You 前一页，backup 前必须有 \appendix。
  • 每个模式都有自己的 section gates。
Spotlight 04

主技能变成 compact router

  • 创建技能主文件保持 220 行以内，避免把所有知识一次塞进上下文。
  • 详细流程、模式、图像策略、验证闸门都拆到 references。
  • 未来 agent 可以按需加载，而不是被长文档牵着跑。

验证与交付闸门

这个技能框架的底线是：不要靠主观感觉宣布 deck 合格。每次交付都要有命令、视觉检查和模式审查证据。

核心命令

./build.sh deck-name python tools/validate_deck.py static deck.tex --mode MODE python tools/check_layout.py deck.tex build/deck.log --advise

硬规则摘要

  • 禁止 overlays：\pause、\onslide、\only、\uncover。
  • 每页最多 3 个彩色盒子。
  • References 在 Thank You 前一页；backup slides 在 \appendix 后。
  • 所有含 block 的页面必须 visual-check，因为日志可能漏掉盒内溢出。
